# Merge Report — Bhumika's UI/UX Design Studio → Ishrat's SDLC Platform

**Date:** 2026-07-10
**Base repository:** `/Users/ishratbhullar/Desktop/SDLC` (Ishrat's, production)
**Merged from:** `final_sdlc_platform_merged.zip` (Bhumika's, earlier fork of the same repo, developed independently since)

## Summary

Three research passes (parallel codebase comparisons) established that Ishrat's backend architecture — modular per-agent packages (`agents/<name>/{agent,prompts,schemas}.py`), the `AgentFactory` registry, pipeline checkpoint-resume, provider-chain LLM service, live code-gen streaming, and a working backend-driven UI style-selection approval flow — is a strict superset of Bhumika's, which predates that refactor entirely. Bhumika's real, valuable contribution was on the **frontend**: a much richer "AI Design Studio" UX for the UI/UX agent (wizard, live canvas, chat, design-token manager, flow diagrams) plus app-shell infrastructure (theme, agent-status polling, workflow coach, autosave indicator). Her Studio was almost entirely client-simulated — its one new endpoint (`POST /generate/uiux-screens`) didn't exist on either backend, and pipeline handoff/status was faked via `localStorage`/`CustomEvent` rather than real state.

This merge ports Bhumika's Studio UX and app-shell components into the base repo, **rewiring every simulated piece to the real backend** (existing endpoints wherever possible, two small additive extensions where genuinely needed), while leaving the backend's architecture untouched as the source of truth.

## Documented scope limits (not bugs — deliberate engineering decisions)

1. **Dark-only theme.** `ThemeProvider`/`useTheme` are mounted for API availability, but `theme` is hardcoded to `'dark'` and `setTheme` is a no-op. The base repo's `tailwind.config.js` has no `darkMode: 'class'` strategy and no light-mode color tokens — every existing component hardcodes the dark palette directly (`bg-dark-bg`, `text-text-primary`, etc.). Building a full light theme was out of scope for this merge and a real regression risk across dozens of components; revisit as a separate, deliberate initiative if light mode is wanted.
2. **No pipeline stop / no live streaming.** Neither backend (FastAPI nor the Node/Express layer) has a `POST /build/stop` endpoint or an SSE pipeline-stream endpoint. `GlobalLoadingContext`'s `canStopPipeline` is always `false` and `GlobalOrchestratorOverlay` has no "Halt Build" button — rather than fabricate a stop that doesn't happen server-side, or fake stream chunks, the overlay shows real state via polling `GET /projects/{id}/pipeline-status` only.
3. **AutoSaveIndicator is client-side-only draft state**, scoped to the new Studio tab's canvas edits, not a general 12-workspace auto-save system. The backend has no draft-artifact concept to sync against.
