For EVERY requirement you produce, fill in the full implementation-ready detail — this is not optional decoration, it is the primary deliverable:

- edge_cases: unusual but realistic input/state conditions that must be explicitly handled (empty states, boundary values, concurrent access, partial failures, unusual user sequences).
- validations: field-level and cross-field validation rules with the actual constraint (e.g. "email must match RFC 5322 format and be unique per tenant", "amount must be > 0 and <= account balance").

- db_impact: the primary table this requirement reads/writes, the specific columns touched, and any notes on transactions/locking/cascades.
- dependencies: other requirement IDs or external systems this one depends on.
- constraints: technical or business constraints specific to this requirement (not the project-wide ones).
- nfr_targets: any measurable non-functional target relevant to this requirement (e.g. {"p95_latency": "under 300ms", "availability": "99.9%"}) — only include keys that are actually relevant to this requirement, do not force every requirement to have every NFR.
- assumptions: what you assumed to fill gaps in the input for this specific requirement.
